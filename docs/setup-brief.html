Browser → Railway → App → Atlas

Hosting · Railway

How the deploy is wired

triggerdeploy
Push to main on GitHub → Railway rebuilds and redeploys automatically. No manual step. Rollback = redeploy a prior build.
builddocker
A 3-stage Docker build: compile the React site → bundle it into the backend and package one jar → copy that jar onto a slim Java runtime image.
edgenetworking
Railway terminates TLS, assigns the public HTTPS domain, and injects the listen port. The app binds $PORT, not a hardcoded number.
runtimeprofile
Runs under Spring's prod profile, which refuses to boot without a real auth secret. Railway probes /actuator/health to gate traffic cutover.
configsecrets
All secrets are Railway variables, never committed. Frontend values must exist at build time to bake into the bundle; backend ones are read at runtime.

Monitoring · Sentry → Slack

How errors reach me

backend

500s only, on purpose

Capture is deliberate: only real unhandled server errors (500s) get sent. Expected 4xx (wrong password, 404, validation) never fires an alert, meaning Slack stays signal and not noise.

frontend

Crashes + masked replay

Browser errors come with a session replay, but every field, input, and network body is masked. So I can see what the user did without seeing their data.

routing

New issue → Slack

A Sentry alert rule forwards each new production issue to my Slack, tagged with the exact deploy (git commit) it came from.

privacy

No secrets leave the app

Auth tokens, cookies, and request bodies are stripped in code on both ends before anything ships to Sentry.

The build method · agentic Claude Code

An agent workflow, not just autocomplete

The interesting part isn't that an AI wrote code. It's the structure I put around it: designs got argued out, invariants got pinned as failing tests before any code, and nothing shipped without passing the gate. Each item below is a reusable Claude Code primitive I wired up for this project. Expand one to see how I used it.

The core discipline. The moment a decision states a rule (weights stored as exact decimals, every muscle trained ≥2×/week, a data-sufficiency gate), it becomes a failing test first, then the code. Every bug this project hit was a known hazard that kept recurring until a test pinned it, so the workflow front-loads that test.

Win · autopilot, end to end

A concurrency fix (two people editing the same workout at once) went from a one-line request to a shipped PR on its own: council-decided, test-first, gate-passed, and signed off by a review council.

Win · caught a false alarm

The QA skill caught bot traffic firing false Sentry alerts. The fix was to return the right client error instead of a 500, which killed the Slack flood at its source. A test now pins it so it can't come back.

New capability · talk to the app

Log a workout by saying it

A small local server lets an AI client (Claude Desktop, Claude Code) read and change your workout data in plain language - "log 3×5 bench at 80kg", "how's my squat trending", "am I gaining or cutting". It's a thin layer over the same API the site already uses, so it inherits every guarantee the app already makes. The standard behind it is MCP (Model Context Protocol) - a common way to hand an AI a set of tools.

rides the API

No new data path

Every tool calls the same /api the site does, so per-user isolation comes for free. The server never touches the database directly.

injected identity

Never hardcoded

Who a call acts for is passed in, not baked in. Locally that's one sign-in at startup; the hosted version swaps in a per-request token and nothing else changes.

holds no state

Stateless by design

The server forgets you between calls, so scaling it is just running more copies. The real load lands on the API and database, which already handle it.

The seam that matters. It's a single-user tool today, built so the multi-user hosted version is a plumbing swap, not a rewrite - change how it's reached, add sign-in, and the 21 tools stay untouched. Local-first pays the cheap part now and defers the expensive hosting and auth work until real users justify it.

today · local

Runs on your machine

Talks to your own account over a direct link to the process. No hosting, no sign-in server - the whole thing is one command.

later · hosted

Every user, their own agent

The same tools behind a web address with proper sign-in. A parked, paid next phase - deliberately not built yet, and not in the shipped image.

trustguardrail
For advice, the tools surface the app's own deterministic coaching engine - the energy estimate, the plan - rather than letting the AI invent training numbers. The engine owns the rules; the AI just voices them.
safetywrites
Reads, logging and edits are all exposed. The two destructive tools - delete a workout, end a plan - are flagged so the client asks before it runs them.
verifiedend to end
Proven with a live round-trip: log a set, read it back, confirm the exact-decimal weight survived as text, then clean up. Unit tests run in the build gate. It's a dev tool, so it's deliberately kept out of the deployed container.
Honest limitations in the setup

The edges, stated plainly

These are deliberate, logged trade-offs, not things I missed. None of them block the demo; each is the next thing to harden before real users show up.

▹

Single instance. The rate limiter and in-progress draft state live in the app's memory. That's correct for one running copy, but scaling to several would need a shared store (Redis, say).

▹

Free-tier database. Atlas M0: shared, 512 MB, no SLA, limited automated backup. Fine for a demo, but real data needs a paid tier and a backup schedule.

▹

Broad DB network access. Railway's plan gives no fixed outbound IP, so Atlas currently accepts connections from anywhere. Known thing to tighten.

▹

Minified frontend traces. Sentry source-map upload is off by choice (turning it on would leak a token into the build), so browser traces aren't fully de-minified yet.

▹

No data-migration path. A change to the data shape currently means re-importing into a fresh database. That needs a real migration story before there's live user data worth preserving.

▹

Subscription and cloud-sync deferred. The seam is already in the code, but billing and multi-device sync are a parked next phase, not built.

▹

Conversational access is local-only. The MCP server runs on my machine against my own account. The hosted, multi-user version with sign-in is built for but not yet shipped - same reason as above, it waits on real demand.
